# Безопасность и секреты

## Главное: в Git **не попадают** пароли пользователей

| Режим | Логин/пароль | Что в GitHub |
|-------|--------------|--------------|
| **GitHub Pages + APK** (standalone) | **Нет** — вход не нужен | Только статический код (`out/` собирается в CI) |
| **Сервер** (`npm run dev`, Vercel) | В **базе на сервере**, не в Git | Только исходный код |

Репозиторий: https://github.com/ElenaSamanchuk/vital-coach  
Сайт после деплоя: https://elenasamanchuk.github.io/vital-coach/

---

## GitHub Pages (ваш сценарий)

- Регистрации и паролей **нет**
- Дневник, профиль, анализы — в **localStorage на телефоне**
- В Git уходит только HTML/JS/CSS — **без данных пользователей**
- APK открывает Pages-URL; секреты в APK не зашиты

**Бэкап:** экспорт JSON в Профиле → файл у пользователя на телефоне, не в Git.

## Серверный режим (если позже поднимете Vercel/Railway)

### Пароли в базе — не «шифрование», а **хеш bcrypt**

Пароль **никогда** не хранится текстом. При регистрации:

```
«Qwerty123!» → bcrypt → $2a$12$... (необратимо)
```

При входе сравнивается хеш, исходный пароль в БД не лежит. Это стандарт (лучше, чем простое шифрование с ключом в коде).

Код: `src/lib/auth.ts`, `src/app/api/register/route.ts` (cost factor 12).

### Что **никогда** не коммитить

| Файл | Содержимое |
|------|------------|
| `.env` | `AUTH_SECRET`, строки БД |
| `*.db` / `prisma/dev.db` | пользователи и **хеши** паролей |
| `out/` | сборка |

Уже в `.gitignore`. Перед push:

```bash
npm run check:secrets
```

### Секреты на GitHub (только для серверного деплоя)

**Settings → Secrets and variables → Actions** (или Variables в Vercel):

| Secret | Зачем |
|--------|--------|
| `AUTH_SECRET` | подпись JWT-сессий (`openssl rand -base64 32`) |
| `DATABASE_URL` | PostgreSQL (Neon/Supabase), **не SQLite в Git** |

Для **только Pages** эти секреты **не нужны** — Actions собирает static export без БД.

## Что видно публично на GitHub

- Исходный код приложения
- **Не видно:** `.env`, база, данные пользователей, пароли (их нет в репо)

Даже при **приватном** репозитории Pages-URL всё равно публичный по ссылке.

## Рекомендации

1. **Массовый APK** — оставить standalone без логинов (проще и безопаснее).
2. Перед первым push: `npm run check:secrets`.
3. Не кладите `dev.db` и `.env` в issue/скриншоты.
4. Для сервера — PostgreSQL на хостинге, не SQLite в репозитории.
5. Опционально позже: PIN + шифрование localStorage на телефоне (отдельная задача).

---

## Если пароль всё же попал в Git по ошибке

1. Сменить пароль у всех затронутых пользователей.
2. Сгенерировать новый `AUTH_SECRET`.
3. Удалить файл из истории Git (`git filter-repo` / GitHub support) — хеш в старом коммите всё ещё опасен, если был **plaintext**; bcrypt-хеш в `dev.db` в истории — риск offline-подбора, БД лучше не коммитить вообще.
